# Phase 02.1: Network Stability & Performance - Context

**Gathered:** 2026-03-31 (updated)
**Status:** Ready for planning
**Source:** User research document + interactive discussion

<domain>
## Phase Boundary

Evolve the networking architecture from a full WebRTC P2P mesh to a hybrid SFU relay that can sustain ~20 concurrent players in fast-paced PvP combat. No new gameplay content — purely networking infrastructure.

**Root cause:** The full mesh topology scales O(N²) — at 20 players each client maintains 19 WebRTC connections, sends position updates to 19 peers at 20 Hz, and performs 19 DTLS encryption/decryption operations per tick. This saturates residential upload bandwidth, overwhelms client CPU with crypto operations, and causes progressive latency degradation.

**Phase 2.1 Plan 01 already completed:** 20 Hz tick rate, dirty-checking, Map lookup, perf metrics. These optimizations carry forward into the new architecture.

</domain>

<decisions>
## Implementation Decisions

### Topology — LOCKED
- D-01: Hybrid SFU relay topology — server becomes the central hub; each client opens ONE WebRTC connection to the server (not to every other peer)
- D-02: Server receives, validates, batches, and fans out messages — no full mesh
- D-03: NO headless Phaser on server — clients still run their own physics/rendering
- D-04: Host-authoritative for damage/death validation on the server side
- D-05: WebRTC MUST remain part of the solution (academic requirement)

### Server WebRTC Library — LOCKED
- D-06: Use `node-datachannel` for server-side WebRTC data channel acceptance (lightweight C++ bindings, designed for game servers)

### Serialization — LOCKED
- D-07: MessagePack binary serialization replaces JSON.stringify/JSON.parse on all data channel messages (schemaless, ~40-60% smaller, fast)

### Server Traffic Management — LOCKED
- D-08: Server-side tick batching at 20 Hz — server accumulates received updates within 50ms window, sends one batched message per client per tick
- D-09: Grid-based spatial interest management with distance falloff — nearby players get full-rate updates, distant players throttled to ~5 Hz

### Migration Strategy — LOCKED
- D-10: 5-step incremental migration (each step independently testable):
  1. Server-side WebRTC + SFU forwarding
  2. MessagePack serialization
  3. Server tick batching
  4. Spatial interest management
  5. Server damage validation
- D-11: Avoid full rewrite — build on existing NetworkManager, EVENT_BUS, Socket.IO lobby infrastructure

### What Stays — LOCKED
- D-12: Socket.IO remains for lobby, team assignment, match start signaling
- D-13: Client-side Phaser game loop, physics, rendering unchanged
- D-14: EVENT_BUS architecture — downstream consumers unchanged
- D-15: Dirty-checking on client before sending position updates (already implemented)
- D-16: 20 Hz tick rate (already implemented in Plan 01)

### Agent's Discretion
- Specific spatial grid cell size and update throttle rates
- Internal module structure for the WebRTC relay (single file vs modular)
- Error handling and reconnection strategy for server WebRTC connections
- Whether to add connection quality metrics (jitter, RTT) beyond existing msg/sec counters

</decisions>

<specifics>
## Specific Ideas

- The server relay module should accept WebRTC data channels using node-datachannel and expose unreliable (position) and reliable (spells/events) channels per client
- Spatial grid should use the existing Tiled map dimensions to set grid cell size
- Damage validation should mirror the existing host-authoritative pattern from PROJECT.md but move it server-side instead of one privileged client

</specifics>

<deferred>
## Deferred Ideas

- Full authoritative server with headless Phaser — possible Phase 5+ if SFU relay proves insufficient
- FlatBuffers serialization — overkill for ~20 players; revisit if scaling beyond 30
- TURN server deployment — SFU topology eliminates most NAT issues; only needed if server is behind NAT
- Client-side prediction and server reconciliation (full) — only needed with authoritative server
- New spells (Ice, Wind, Thunder) — Phase 3
- PvP combat mechanics — Phase 4
- Match loop — Phase 5

</deferred>
