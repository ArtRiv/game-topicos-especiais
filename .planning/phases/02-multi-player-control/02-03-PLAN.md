# Plan 02-03: GameScene Deterministic Team Tinting

**Goal:** Give each remote player a consistent, deterministic tint color based on their team or their fixed index in `matchConfig.players` — so all clients assign the same color to the same player regardless of message arrival order.
**Estimate:** S
**Depends on:** 02-01 (requires `team` field on `PlayerInfo`)
**Delivers:** PLR-01 (no hardcoded role assumptions), PLR-03 (remote players render correctly on all clients)

---

## Context

Currently in `GameScene.#onRemotePlayerUpdate`, the tint assigned to a new remote player is:
```typescript
const slotIndex = this.#remotePlayers.size + 1;
const tint = GameScene.#PLAYER_TINT_PALETTE[slotIndex % GameScene.#PLAYER_TINT_PALETTE.length];
```
`#remotePlayers.size` depends on the order that `NETWORK_PLAYER_UPDATE` messages happen to arrive. With 3+ players, different clients can assign different colors to the same player — a problem that becomes visible to users and also makes debugging harder.

The fix: expose `NetworkManager.#matchPlayers` (which already holds `PlayerInfo[]` including `team`) via a public getter, then look up the player's index in that array for a stable tint. Players on Team A get blue (`0x0055ff`), Team B get red (`0xdd2200`). Unassigned or offline (no matchConfig) falls back to the existing palette indexed by stable player-array position.

**Timing note**: `NETWORK_LOBBY_STARTED` fires before `GameScene` exists (lobby → preload → game). Therefore, a listener inside `#setupNetworking()` would miss the event. The correct approach is to read `nm.matchPlayers` directly at the time a remote player is first created (inside `#onRemotePlayerUpdate`), since `NetworkManager.#matchPlayers` is already populated from the `lobby:started` socket event.

Add the helper method `#resolveRemotePlayerTint` to the networking section of `GameScene` (after `#setupNetworking`):
```typescript
/**
 * Returns a deterministic tint for a remote player.
 * Uses team-based colors when team data is available from matchConfig.
 * Falls back to stable-index palette when team is unassigned or matchConfig is unavailable.
 */
#resolveRemotePlayerTint(playerId: string): number {
  let nm: NetworkManager | null = null;
  try { nm = NetworkManager.getInstance(); } catch { /* offline */ }

  if (nm) {
    const matchPlayers = nm.matchPlayers;
    const playerIndex = matchPlayers.findIndex((p: PlayerInfo) => p.id === playerId);
    if (playerIndex !== -1) {
      const info = matchPlayers[playerIndex];
      if (info.team === 0) return 0x0055ff;  // Team A — blue
      if (info.team === 1) return 0xdd2200;  // Team B — red
      // Unassigned: use stable array index offset by 1 to skip palette[0] = 0xffffff (white).
      // White is reserved as the local player's identity color; a remote player that happens
      // to be at matchPlayers[0] must never receive white or they are visually indistinguishable
      // from the local mage.
      return GameScene.#PLAYER_TINT_PALETTE[(playerIndex + 1) % GameScene.#PLAYER_TINT_PALETTE.length];
    }
  }

  // Offline / no matchConfig: fall back to insertion-order slot (existing behavior)
  const slotIndex = this.#remotePlayers.size + 1;
  return GameScene.#PLAYER_TINT_PALETTE[slotIndex % GameScene.#PLAYER_TINT_PALETTE.length];
}
```

Note: `#PLAYER_TINT_PALETTE[0]` is `0xffffff` (local player / white). Remote players come from `matchPlayers` at index > 0 relative to the local player, so the array-index lookup keeps white for local and assigns distinct colors to remotes.
