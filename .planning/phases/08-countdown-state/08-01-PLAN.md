<objective>
Replace the Phase 7 STUB (`setTimeout(50ms)` auto-advance from COUNTDOWN→ACTIVE) with the real server-driven countdown timing: 4 per-second `match:countdown-tick` broadcasts (`3 → 2 → 1 → FIGHT!`) followed by the genuine `LOADING → ... → ACTIVE` transition at t+3500 ms. Make `lobby:start` idempotent (CR-02 fix from `07-REVIEW.md`) so a host double-click cannot spawn a second `GameRoom` with a second countdown timer racing the first. Store all timer handles on `GameRoom` so disconnect / room-empty / state-change-out-of-COUNTDOWN cleanly cancels them (WR-07 fix from `07-REVIEW.md`).

Purpose: Phase 8 of v1.2 (Match Lifecycle & Event Polish) — Plan 08-01 covers the entire server-side delta for LFC-08 (overlay drive by per-second tick broadcasts) and the server half of LFC-09 (simultaneous unlock via a single `match:state-changed: ACTIVE` broadcast). The client cinematic + input lock + overlay rendering live in Plan 08-02. CR-02 (lobby:start non-idempotent) and WR-07 (free-floating setTimeout) from `07-REVIEW.md` are folded in because the new real timer makes both bugs strictly worse if left in place (double-click → two interleaved countdown streams; disconnect mid-countdown → late-firing orphaned timers).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → server (socket.io `lobby:start`) | Untrusted: host or impostor may emit twice; the server cannot trust this to be once-per-match. |
| internal scheduler → io.emit | Pending `setTimeout` callbacks can fire after the room has been destroyed or replaced; the callback identity must be verified before emitting. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-08-01 | Denial of Service | `lobby:start` handler (server.ts:89-103) | mitigate | Defense in depth: (a) `lobbyManager.startLobby()` rejects when `lobby.status !== 'waiting'` (lobby-manager.ts); (b) `if (gameRooms.has(lobby.id)) return;` at the top of the server.ts handler. A double-clicked or replayed Start cannot spawn a second `GameRoom` with a second timer. (CR-02) |
| T-08-02 | Tampering / use-after-free | `setTimeout` callbacks in `startCountdown` | mitigate | Every callback captures `room` by reference and verifies `gameRooms.get(lobbyId) === capturedRoom && room.state === 'COUNTDOWN'` before emitting/transitioning. Handles are stored on `GameRoom.#countdownHandles` and cleared on `transitionTo` out of COUNTDOWN or `removePlayer` when the room becomes empty. (WR-07) |
| T-08-03 | Spoofing | inbound `match:countdown-tick` from a peer | accept | The event is server-emit-only; clients never emit it. Even if a malicious peer forges it on their own socket, the server has no `socket.on('match:countdown-tick', ...)` handler, so the event is silently dropped. No new attack surface. |
| T-08-04 | Tampering (inherited from Phase 7 WR-02) | `match:loaded` handler payload validation | accept (scoped) | Out of scope for Plan 08-01 — already filed under Phase 7 WR-02. Plan 08-01 preserves the existing `({ lobbyId }: MatchLoadedPayload)` shape; payload-shape validation is a separate hardening pass. |
</threat_model>
